# 医保飞检智能检索项目架构与前期开发任务

生成日期：2026-07-12

## 1. 项目目标

建设一个面向医保飞检专家的轻量智能检索系统，先完成现场数据库适配、核心字段映射、国家医保规则匹配、受控 SQL 查询、结果回传 H5 和基础测试闭环。

项目第一阶段的核心目标不是建设完整监管平台，而是让专家能够：

1. 连接现场只读数据库；
2. 选择核心业务表；
3. 完成少量核心字段映射；
4. 将国家医保规则和项目清单与现场数据对应；
5. 通过文字或语音提出查询问题；
6. 生成结构化查询条件；
7. 经专家确认后生成受控 SQL；
8. 执行查询；
9. 将汇总、明细、图表和导出文件返回 H5。

## 2. 明确不做

本阶段只明确不做以下三项：

- 不做多数据库通用平台；
- 不做映射审批流；
- 不做复杂大模型编排。

除上述三项外，与现场接入、规则执行、查询准确性、结果回传和测试直接相关的必要功能均纳入本项目。

## 5. 国家医保规则侧准备

### 5.1 国家规则标准化

将国家医保智能监管“两库”公开附件统一转换为以下结构：

```yaml
rule_id:
rule_name:
rule_type:
target_item_codes:
target_item_names:
condition:
threshold:
time_window:
effective_from:
effective_to:
logic_text:
policy_basis:
source_batch:
source_file:
source_sheet:
source_row:
```

### 5.2 规则最低字段需求矩阵

每类规则必须定义执行所需最小字段。

示例：

```yaml
duplicate_charge:
  required:
    - settlement_id
    - item_code
    - item_name
    - quantity
    - amount
    - fee_time
  recommended:
    - refund_flag
    - department_name

frequency_limit:
  required:
    - settlement_id
    - item_code
    - quantity
    - fee_time
  recommended:
    - department_name
    - refund_flag

insurance_flag_check:
  required:
    - item_code
    - insurance_flag
    - amount

price_check:
  required:
    - item_code
    - unit_price
    - quantity
    - amount
    - fee_time
```

### 5.3 首批执行规则

第一阶段优先支持：

- 项目是否医保；
- 项目重复收费；
- 数量或频次超限；
- 单价异常；
- 金额异常；
- 项目共现或互斥；
- 按科室统计；
- 按项目统计；
- 按机构统计；
- 按月份统计。

后续可在现场字段具备时增加：

- 性别限制；
- 年龄限制；
- 机构等级限制；
- 就医方式限制；
- 诊断和适应症限制；
- 支付疗程限制。

## 6. 现场数据库映射工具

### 6.1 数据源连接

第一阶段只支持现场确定的一种数据库类型，SQLite作为本地测试数据库。

功能：

- 数据库地址、端口、数据库名、Schema；
- 只读账号和密码；
- 连接测试；
- 读取表和字段元数据；
- 连接超时设置；
- 不保存真实密码到 Git。

### 6.2 核心表选择

系统列出数据库表，由用户选择或根据系统推荐确认：

- 结算主表；
- 费用明细表；
- 项目字典表；
- 科室或机构表；
- 退费或冲销表，如存在。

不要求系统自动推断全部表关系，少量关键表由现场人工确认。

### 6.3 核心字段映射

第一阶段核心字段：

```text
settlement_id
encounter_id（可选）
institution_code（可选）
institution_name（可选）
department_code（可选）
department_name
item_code
item_name
insurance_flag
unit_price
quantity
amount
fee_time
refund_flag（可选）
reversal_flag（可选）
```

扩展规则需要时再映射：

```text
gender
age
birth_date
institution_level
visit_type
diagnosis_code
insurance_type
```

### 6.4 字段推荐方式

推荐顺序：

1. 字段名精确匹配；
2. 字段别名匹配；
3. 字段注释匹配；
4. 少量脱敏样例辅助；
5. 大模型对难以判断的候选字段进行语义排序；
6. 人工最终确认。

大模型只做轻量候选推荐，不直接写入正式映射。

### 6.5 项目编码匹配

实现：

1. 国家编码与现场编码精确匹配；
2. 项目名称精确匹配；
3. 标准化名称匹配；
4. 别名和简称匹配；
5. 少量未匹配项人工选择；
6. 必要时大模型提供候选排序。

状态：

```text
exact
recommended
confirmed
pending
unmatched
invalid
```

第一阶段不自动处理复杂的一对多、多对一决策，出现时由现场人工确认。

### 6.6 值域确认

不建设复杂值域映射平台，但对以下关键字段提供简单值域查看和人工确认：

- 是否医保；
- 退费或冲销标识；
- 门诊、住院、急诊类型；
- 性别；
- 机构等级。

## 7. 现场探测与验证

完成映射后，执行轻量只读探测，不进行全库深度扫描。

探测内容：

- 关键字段是否存在；
- 核心字段空值率；
- 主表与费用明细能否关联；
- 项目编码与项目字典匹配率；
- 日期范围；
- 数量、单价、金额基本范围；
- 是否存在负金额；
- 退费和冲销如何表达；
- 是否存在明显重复上传；
- 科室和机构字段是否可用。

输出：

```json
{
  "mapping_status": "ready | partially_ready | blocked",
  "available_rules": [],
  "missing_fields": [],
  "mapping_issues": [],
  "probe_summary": {}
}
```

## 9. 查询理解与 H5 交互

### 9.1 文字与语音输入

H5支持：

- 文字输入；
- 浏览器语音转文字；
- 用户修改转写内容；
- 提交查询问题。

### 9.2 查询理解

系统从问题中提取：

- 规则类型；
- 项目名称或编码；
- 时间范围；
- 科室或机构；
- 是否医保；
- 价格、数量、金额条件；
- 聚合口径；
- 输出要求。

无法确定的信息必须返回缺失字段，不得使用固定默认值代替。

### 9.3 专家确认

执行前向专家展示：

```text
采用规则
目标项目
本地项目编码
时间范围
科室或机构范围
判断条件
聚合口径
金额口径
缺失或不确定项
```

专家可临时修改和补充。

## 10. 受控 SQL 生成

采用：

```text
国家规则
+ 结构化查询条件
+ 现场映射配置
+ 参数化 SQL 模板
= 现场可执行 SQL
```

要求：

- 仅允许 SELECT；
- 表和字段白名单；
- 参数绑定；
- 强制时间范围；
- 最大返回行数；
- 查询超时；
- 禁止无条件全表扫描；
- 禁止用户原始 SQL 直接执行；
- 患者标识默认脱敏；
- 保存最终 SQL 和参数。

## 11. 查询执行和效率优化

第一阶段不对目标数据库做全量数据抽取，只查询规则所需最少字段和目标项目数据。

查询流程：

```text
确定目标规则
        ↓
获取目标项目编码集合
        ↓
映射到现场项目编码
        ↓
确定规则所需最小字段集
        ↓
限定时间、机构、科室和项目范围
        ↓
数据库端筛选和聚合
        ↓
仅返回命中明细和汇总
```

禁止：

```sql
SELECT *
```

优先：

```sql
SELECT
  settlement_id,
  department_name,
  item_code,
  item_name,
  insurance_flag,
  unit_price,
  quantity,
  amount,
  fee_time
```

查询分级：

- 快查询：直接执行；
- 普通查询：后台任务；
- 大查询：限制范围或分批执行。

第一阶段可继续使用轻量任务持久化；真实并发压力明确后再增加 Redis 和独立 Worker。

### 13.6 最小安全测试

## 17. 最终结论

本阶段项目主线为：

```text
国家医保规则标准化
        ↓
现场核心表和字段映射
        ↓
少量人工修正
        ↓
生成映射配置
        ↓
规则驱动受控SQL
        ↓
只查询必要字段和目标项目
        ↓

```

项目强调实用性：能够人工确认的少量步骤不做过度自动化，优先保证现场接得上、规则查得准、金额算得对、结果回得来。
