# 医保飞检智能检索 H5 MVP 架构 v2

## 1. 架构调整目标

本版本在原有架构基础上明确两条主线：

```text
规则库 = 结构化入库 + 代码化执行 + SQL 模板绑定
医保知识库 = 文档解析 + 切片 + 向量检索 + 政策依据引用
```

也就是说：

- 规则库不是普通 RAG 文档库，而是可执行、可测试、可审计的业务规则代码。
- 医保库是 RAG 知识库，用于政策理解、依据引用、术语解释和辅助匹配。
- 两个压缩包都要使用，但使用方式不同。

## 2. 总体业务链路

```text
专家自然语言问题
    ↓
医保知识库 RAG 检索
    ↓
识别医保概念、项目名称、政策依据、近义词
    ↓
匹配代码化规则库
    ↓
生成结构化查询条件
    ↓
专家确认项目、规则、时间范围、口径
    ↓
调用规则执行器
    ↓
套用受控 SQL 模板
    ↓
查询医保数据库只读副本
    ↓
返回汇总、明细、图表、Excel
    ↓
附带政策依据引用和执行日志
```

## 4. 规则库代码化设计

### 4.1 规则来源

第一优先级来源：

- 国家医保智能监管两库官方公开附件第一至十八批。

后续来源：

- 本省已经明确可执行的监管规则。
- 专家沉淀的金标准规则。
- 手工确认后的规则补充。

### 4.2 规则处理流程

```text
Excel / 人工规则
    ↓
字段标准化
    ↓
规则类型识别
    ↓
结构化规则入库
    ↓
绑定规则执行器
    ↓
绑定 SQL 模板
    ↓
金标准案例回归测试
```

### 4.3 规则最小字段

```yaml
rule_id:
rule_code:
rule_name:
rule_type:
target_type:
target_codes:
target_names:
applicable_scope:
  region:
  institution_level:
  encounter_type:
  insurance_type:
condition:
  metric:
  operator:
  threshold:
  time_window:
exclusions:
sql_template_id:
executor_class:
policy_basis:
source_file:
version:
status:
effective_from:
effective_to:
```

### 4.4 一期规则执行器

| 规则类型 | 执行器 | 典型来源 |
|---|---|---|
| 性别限制 | GenderLimitRule | 药品区分性别、医疗服务项目区分性别 |
| 年龄限制 | AgeLimitRule | 儿童专用、限儿童、限年龄 |
| 频次限制 | FrequencyLimitRule | 限定频次、周期超频次 |
| 重复收费 | DuplicateChargeRule | 医疗服务项目重复收费 |
| 支付疗程限制 | TreatmentCourseLimitRule | 限支付疗程 |
| 机构级别限制 | InstitutionLevelLimitRule | 药品限医疗机构级别 |
| 就医方式限制 | EncounterTypeLimitRule | 药品限就医方式 |
| 适应症限制 | IndicationLimitRule | 药品限适应症 |

规则执行器输入：

```text
结构化查询条件 + 规则参数 + 字段映射 + 数据范围
```

规则执行器输出：

```text
参数化 SQL + 查询口径 + 输出字段 + 结果解释模板
```

## 5. 医保 RAG 知识库设计

### 5.1 知识库来源

- 本省医保政策文件。
- 医疗服务价格文件。
- 医保支付政策。
- 集采政策。
- 医保目录说明。
- 政策通知正文和附件说明。

### 5.2 RAG 处理流程

```text
PDF / DOCX / XLSX / WPS / ET / DOC / RAR
    ↓
文件台账登记
    ↓
格式转换或解压
    ↓
正文和表格抽取
    ↓
元数据提取
    ↓
政策条款切片
    ↓
Embedding 向量化
    ↓
RAG 检索
    ↓
返回政策依据、文号、原文片段、来源文件
```

### 5.3 文档元数据

```yaml
document_id:
title:
doc_no:
issuer:
publish_date:
effective_date:
region:
source_type:
source_file:
content_hash:
parse_status:
```

### 5.4 切片元数据

```yaml
chunk_id:
document_id:
chunk_index:
section_title:
content:
embedding:
page_no:
metadata:
```

### 5.5 RAG 返回要求

每次 RAG 检索不能只返回文本，必须返回：

- 政策标题；
- 文号；
- 发文日期；
- 执行日期；
- 原文片段；
- 来源文件；
- 页码或附件名；
- 置信度。

## 6. 文档格式处理策略

所有文件都纳入资产库，不丢弃特殊格式。

| 格式 | 是否使用 | 处理方式 |
|---|---:|---|
| XLSX/XLS | 是 | 优先结构化；根据内容进入规则库、项目清单或 RAG |
| PDF | 是 | 解析正文和表格，进入医保 RAG |
| DOCX | 是 | 解析正文，进入医保 RAG |
| DOC | 是 | 转 DOCX/PDF 后解析 |
| WPS | 是 | 转 DOCX/PDF 后解析 |
| ET | 是 | 转 XLSX 后分类处理 |
| RAR | 是 | 解压后重新登记和分类 |
| MD | 是 | 作为架构和项目文档资产 |

## 7. 开发实施顺序

### 阶段一：资料盘点与资产分流

1. 登记架构 MD、国家两库 ZIP、本省政策 ZIP。
2. 解压两个 ZIP。
3. 建立文件台账。
4. 统计格式、大小、路径、来源包、内容类别。
5. 初步分类为规则库、医保 RAG、项目清单、待转换、待人工确认。

### 阶段二：规则库代码化

1. 抽取国家两库 Excel 表头。
2. 设计统一规则字段。
3. 建立 rule_definitions、rule_items、rule_parameters。
4. 实现 5 至 10 类规则执行器。
5. 建立 SQL 模板。
6. 准备金标准案例。
7. 每次规则代码变更后执行回归测试。

### 阶段三：医保 RAG 知识库

1. 解析本省政策 PDF、DOCX、XLSX、XLS。
2. 转换 WPS、ET、DOC、RAR 后继续处理。
3. 提取标题、文号、发文日期、执行日期、来源文件。
4. 文本切片。
5. 向量化入库。
6. 实现政策检索接口。
7. 建立 RAG 检索效果测试集。

### 阶段四：数据库查询闭环

1. 获取医保数据库只读连接。
2. 固定核心表和字段映射。
3. 将规则执行器输出接入 SQL 模板。
4. 实现 SQL 安全校验。
5. 实现任务队列、Worker、分页和导出。

### 阶段五：H5 验收闭环

1. 新建查询。
2. 条件确认。
3. 展示规则和政策依据。
4. 任务列表。
5. 汇总、图表、明细。
6. Excel 导出。

## 8. 一期验收标准

- 所有原始资料进入资产台账。
- 国家两库至少完成 3 类规则代码化。
- 本省政策文件至少完成一批 RAG 入库。
- 专家问题可以匹配项目、规则和政策依据。
- 专家确认后可以生成结构化查询条件。
- 规则执行器可以生成受控 SQL。
- 查询过程和版本可追溯。
