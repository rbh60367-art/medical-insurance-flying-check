# 医保飞检智能检索前端架构方案

生成日期：2026-07-13

## 1. 前端定位

前端不应只做一个简单查询页，而应设计为“医保飞检工作台”。

系统面向三类使用者：

1. 飞检专家：提出问题、确认规则和查询条件；
2. 业务审核人员：查看疑点明细、复核证据、导出结果；
3. 管理领导：查看总体风险、规则命中情况和机构排名。

前端核心目标：

```text
专家负责查
审核人员负责复核
领导负责看结果
```

## 4. 智能查询页

### 4.1 页面目标

给飞检专家使用，用于提出问题、确认规则、生成查询条件和执行任务。

### 4.2 页面区域

```text
顶部：文字 / 语音输入框
中部左：候选规则
中部右：政策依据 / 知识库依据
下方：生成的查询条件
底部：确认并生成 SQL / 执行查询
```

### 4.3 用户流程

```text
输入问题
  ↓
系统识别规则和项目
  ↓
返回候选规则、政策依据、查询条件
  ↓
专家确认
  ↓
生成受控 SQL
  ↓
执行查询
  ↓
跳转或刷新结果页
```

### 4.4 关键字段

```text
question
recognized_conditions
candidate_rules
policy_hits
generated_conditions
generated_sql
task_id
status
```

### 5.2 核心指标

```text
疑点明细数
涉及机构数
```

### 5.3 图表

### 5.4 示例展示


## 6. 规则命中汇总页

### 6.1 页面目标

按规则维度展示本次检查命中情况，帮助业务负责人快速判断哪些规则风险最高。

### 6.2 表格字段

```text

涉及科室数

```

### 6.3 筛选条件

```text
规则类型
风险等级
金额区间
机构
时间范围
```

### 6.4 示例表格

| 规则 | 类型 | 命中明细数 | 疑点金额 | 涉及机构 | 风险等级 |
|---|---|---:|---:|---:|---|
| 急诊监护费 + 急诊诊查费重复收费 | 重复收费 | 128 | 35,200 | 4 | 高 |
| 某项目单价超标准 | 单价异常 | 76 | 18,900 | 2 | 中 |
| 非医保项目纳入医保支付 | 医保属性 | 43 | 12,500 | 3 | 高 |

## 7. 机构 / 科室分析页

### 7.1 页面目标

定位问题集中在哪些机构、科室和项目。

### 7.2 展示内容

```text

科室排名
项目排名
机构-规则交叉表
按月趋势
```

### 7.3 适用场景

领导或专家想知道：

```text
哪家机构问题最多？
哪个科室疑点金额最高？
哪些收费项目最容易命中规则？
某类问题是否集中在某个时间段？
```

## 9. 现场映射配置页

### 9.1 页面目标

这是当前前期开发中最重要的页面之一，用于现场数据库适配。

### 9.2 页面步骤

```text
Step 1 数据源连接
Step 2 选择核心表
Step 3 字段映射
Step 4 项目编码匹配与探测
```

### 9.3 字段映射表

| 标准字段 | 推荐现场字段 | 置信度 | 状态 | 人工修改 |
|---|---|---:|---|---|
| settlement_id | jslsh | 0.82 | 推荐 | 可修改 |
| item_code | xmbm | 0.82 | 推荐 | 可修改 |
| amount | je | 0.82 | 推荐 | 可修改 |
| fee_time | fssj | 0.82 | 推荐 | 可修改 |

### 9.4 探测结果

```text
映射状态：ready / partially_ready / blocked
可执行规则数
缺失字段
字段空值率
主表-明细表关联情况
项目匹配率
```

### 9.5 设计边界

本阶段不做复杂映射审批流。少量不确定字段允许现场人工临时修改。

## 15. 总结

前端不应只做“查询工具”，而应做成“医保飞检工作台”。

核心思路：

```text
专家通过智能查询发起任务
系统通过规则和知识库生成检索条件
专家确认后执行受控 SQL
审核人员查看疑点明细
领导查看汇总看板
```
