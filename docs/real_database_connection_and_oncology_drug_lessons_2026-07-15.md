# 医保飞检真实数据库打通与肿瘤特药数据源追踪经验

生成日期：2026-07-15

> 本文为脱敏版经验沉淀，不记录数据库密码、真实服务器地址、真实 schema 名、患者姓名、住院号、结算 ID、就诊 ID 等敏感信息。用于说明项目后续如何从 MVP 原型走向真实医保业务落地。

## 1. 本次经验的价值

本次验证说明，医保飞检系统落地时，最大的难点不只是 AI 问答，而是把真实业务数据链路打通：

```text
网络可达
-> 数据库可登录
-> 找到正确医院 schema
-> 确认主查询表
-> 对齐字段和编码
-> 追踪患者费用明细
-> 关联结算、就诊、诊断
-> 对账 Excel 口径
-> 批量形成可复核结果
```

因此，系统后续要把“真实数据源确认”和“检查口径对齐”作为工作流的一部分，而不是只提供一个查询框。

## 2. 关键业务认识

### 2.1 数据库打通不等于业务打通

能连接数据库，只说明技术链路可用；真正业务可用还需要确认：

- 当前任务对应哪家机构；
- 对应哪个业务 schema；
- 费用明细主表是哪张；
- 结算、就诊、诊断表如何关联；
- Excel 或人工报表使用了什么过滤口径；
- 金额字段到底取总金额、医保范围金额，还是基金支付金额。

### 2.2 查错 schema 会导致业务误判

同一个药品或项目可能在目录表存在，但在某个 schema 的费用明细中没有患者记录。早期如果只查一个 schema，容易误判为“没有真实使用记录”。

正确做法是：

```text
先扫描当前账号可访问 schema / 表
-> 识别医院代码、机构名称、表命名规律
-> 找到与 Excel 数据来源一致的 schema
-> 再做费用明细验证
```

### 2.3 肿瘤特药查询不是普通关键词查询

肿瘤重点特药查询需要同时穿透：

- Excel 特药清单；
- 国家医保编码；
- 本地项目编码；
- 费用明细；
- 结算主表；
- 就诊主表；
- 诊断明细；
- 原发恶性肿瘤诊断口径；
- 金额统计和对账口径。

这类任务更适合做成“批量工作流”，不适合逐药手工查询。

## 3. 推荐系统工作流

### 3.1 数据源确认

输入：数据库连接、机构线索、Excel 来源说明。

系统动作：

- 读取当前账号可访问表；
- 识别候选 schema；
- 标记费用、结算、就诊、诊断等核心表；
- 输出推荐数据源档案。

输出：数据源档案。

```text
机构：已脱敏医院
数据库类型：Oracle
主 schema：已脱敏
费用明细主表：FEE_LIST_D_xxxx 类表
结算主表：SETL_D_xxxx 类表
就诊主表：MDTRT_D_xxxx 类表
诊断表：SETL_DIAG_LIST_D_xxxx 类表
```

### 3.2 字段映射

输入：现场表字段。

系统动作：

- 将现场字段映射为系统标准字段；
- 标记必填字段是否缺失；
- 给出人工确认项。

输出：字段映射结果。

核心字段包括：

```text
settlement_id
encounter_id
patient_id
institution_code
institution_name
department_name
item_code
item_name
local_item_code
local_item_name
quantity
unit_price
amount
insurance_scope_amount
fund_pay_amount
fee_time
gender
age
diagnosis_code
diagnosis_name
main_diagnosis_flag
diagnosis_order
```

### 3.3 肿瘤特药清单导入

输入：Excel 重点特药清单。

系统动作：

- 解析药品名称；
- 解析国家医保编码；
- 构造 drug_map 临时映射；
- 统一做批量 JOIN。

输出：标准化药品映射表。

### 3.4 批量检索

不要逐药跑 33 次，应构造药品映射后一次性查询：

```text
drug_map
JOIN 费用明细
JOIN 结算主表
JOIN 就诊主表
LEFT JOIN 诊断明细
```

输出：药品级汇总、患者级明细。

### 3.5 原发恶性肿瘤诊断筛选

推荐口径：

```sql
DIAG_CODE LIKE 'C%'
AND DIAG_NAME LIKE '%恶性肿瘤%'
AND DIAG_NAME NOT LIKE '%继发%'
```

诊断优先级：

```text
优先取主诊断
否则取诊断顺序最靠前的原发恶性肿瘤诊断
```

### 3.6 Excel 对账

数据库统计金额和 Excel 金额可能接近但不完全一致。系统需要支持口径对齐：

- 是否排除退费 / 冲销；
- 是否只看有效记录；
- 是否只看已结算；
- 是否限定门诊 / 住院；
- 是否只统计医保范围金额；
- 是否只统计基金支付金额；
- 是否按国家编码精确匹配；
- 是否允许名称扩展匹配；
- 时间边界是否一致。

输出：对账结果。

```text
Excel 金额
数据库金额
差异金额
差异率
可能原因
```

## 4. AI 应该落地的位置

这次经验说明，AI 不能替代数据库事实和规则口径，但可以嵌入以下环节：

1. 根据自然语言生成检查任务；
2. 辅助识别正确数据源和候选表；
3. 根据字段名推荐标准字段映射；
4. 根据药品清单生成批量查询模板；
5. 解释金额差异可能来自哪些过滤口径；
6. 根据结果生成复核摘要和汇报说明。

正确关系是：

```text
业务是主线
规则是骨架
数据是落点
AI 是助手
```

## 5. 对当前系统的修改方向

当前系统已经具备：

- 规则库检索；
- 政策 RAG；
- 现场字段映射；
- 受控 SQL；
- 疑点结果；
- 检查看板。

根据本次真实数据库验证，下一步应补强：

1. 数据源档案管理；
2. Oracle 现场 schema / 表识别；
3. 肿瘤特药批量查询模板；
4. 过滤口径配置；
5. Excel 对账看板；
6. 患者级明细导出和复核。
